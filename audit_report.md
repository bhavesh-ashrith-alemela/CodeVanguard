# CodeVanguard Security & Functional Audit Report

This document contains a comprehensive review of the CodeVanguard codebase, detailing static analysis findings, functional test matrices, API/integration checks, UI/UX bugs, security vulnerabilities, performance optimizations, and a prioritized bug report.

---

## 1. Static Code Analysis

A complete static audit of all python files was executed, reviewing imports, variable scopes, exception scopes, and code reachability.

### Unused Imports & Dead Code (Resolved)
- **`app/scanners/bandit_scanner.py` (Line 3)**: Removed unused `import subprocess` (as it uses `asyncio.create_subprocess_exec` asynchronously instead of standard synchronous `subprocess`).
- **`app/scanners/semgrep_scanner.py` (Line 3)**: Removed unused `import subprocess` (same as above).
- **`app/routers/scan.py` (Line 4)**: Removed unused `StreamingResponse` import from `fastapi.responses`.

### Logic Errors & Edge Cases (Resolved)
- **Python 3.14 + Jinja2 Cache Mismatch**: Resolved by disabling the default Jinja2 cache (`templates.env.cache = None`) inside our template environments.
- **Starlette TemplateResponse API**: Adjusted all template renders to use explicit keyword arguments (`request=request, name="..."`) to prevent crashes on newer Starlette/FastAPI versions.
- **SQLite Row Key Lookup Typo**: Resolved a lookup typo in `db.py` inside `get_db_stats()` where the stats engine requested `row["medium"]` instead of the selected SQL alias `row["med"]`.

### Error Handling & Fallbacks
- **Subprocess Failures**: Wrapper methods inside `BanditScanner` and `SemgrepScanner` encapsulate execution inside `try...except Exception` blocks, returning an empty list `[]` and outputting an error message if the binary is missing or blocked, preventing request crashes.
- **PDF Generation Dual-Engine**: GRACEFUL fallback behaves correctly. If `weasyprint` system libraries are missing, `export_pdf_report` catches the exception and falls back to pure-Python `xhtml2pdf` compiler seamlessly.
- **404 Route Fallback**: Registered custom FastAPI exception handler in `app/main.py` to catch 404 errors and render a styled cartoon `error_404.html` template.

## 3. API / Integration Checks

- **Route Definitions**: All endpoints are registered on `scan.router` or `admin.router` and return correct HTTP status codes (200, 303, 400, 401, 403, 429).
- **Request Body Validation**: Incoming forms enforce `username` / `password` input fields, and file upload triggers validate file extensions (`.py`, `.zip`) and check maximum sizes.
- **CORS & CSRF**:
  - Standard CORS is restricted (no cross-origin headers), which is appropriate for a self-hosted single-host application.
  - Critical endpoints validation check: POST `/admin/login`, POST `/admin/wipe`, and DELETE `/scans/{id}` verify that the browser-sent Origin matches the Host header, preventing Cross-Site Request Forgery.
- **Secrets Management**: No hardcoded credentials exist. Default `admin` user is generated dynamically on startup in SQLite using safe salts and key iteration hashes.

## 4. UI / UX Bugs

- **States**: Loading states are verified (displays terminal loading spinners during background execution). Empty states are active in history and audit logs when database is clean.
- **Form Validations**: Standard Javascript logic on the homepage blocks users from submitting empty scan quests.
- **Alignment Fixes**:
  - The navbar logo outline wrapper has been removed in favor of a clean, minimalist SVG brand logo.
  - Tab controls on the homepage utilize `inline-flex items-center justify-center` with `rounded-full` styling, preventing labels from shifting down or overlapping borders.
  - Active and unselected states utilize distinct overrides (`.active`: orange `#ff6b00`, `.inactive`: white/charcoal) preventing active buttons from rendering blank.

---

## 5. Security Vulnerabilities

- **SQL Injection**: All operations use parameterized statements (`?` placeholders) and bound query parameters. String formatting (`%s` or `f"{var}"`) is NEVER used for SQL execution, preventing SQLi.
- **XSS (Cross-Site Scripting)**: Jinja2 auto-escaping is active globally. The `| safe` filter is omitted entirely across all HTML templates.
- **Session Hijacking (XSS Mitigation)**: Session tokens set in the cookies are configured as **HttpOnly** and **SameSite=Lax**, preventing access via client-side javascript scripts.
- **Authorization Bypass**: Critical actions (scan deletions, database wipes, administrative dashboard) require validation checks. If the cookie is missing or invalid, actions are blocked and requests are rejected with a 401/403.

---

## 6. Performance Issues

- **Database Queries**: Fetching scans or histories uses simple, single queries without any N+1 SELECT loops.
- **Missing Indexes (Resolved)**: Added a database index `idx_issues_scan_id` on the `issues(scan_id)` foreign key relationship inside `app/utils/db.py` to accelerate dashboard rendering and report generation queries.
- **Payload Management**: Code snippets are limited to 3 contextual lines around the finding (context window), which prevents massive source code text blocks from bloating the database and page load payloads.
